# 魔方还原助手 · 产品设计文档

> 平台：AIUI（Rokid Glasses 单色 Micro-LED 眼镜，运行时 Ink/QuickJS）
> 定位：新手魔方还原教练 · 用途：本人开发用工作规格 · 更新：2026-06-17
> 配套：硬件/API 速查 `硬件与API开发参考.md`；逐次改动记录 `开发状态.md`。
> 说明：本文档只讲「要做什么 + 关键决策」，技术细节见附录与配套文档。

---

## 1. 产品定义

**戴上眼镜，把魔方逐面对着镜头拍一下，跟着视野里的箭头一步步转，把打乱的三阶魔方还原。**

面向**完全不会还原魔方的新手**：免手操作（双手始终握着魔方、语音翻页），一次只显示一步，针对你手里这个具体魔方算出专属解法，无需低头看手机教程。

---

## 2. 设计前提：硬件硬约束

三条边界决定了整体架构（能力细节见 `硬件与API开发参考.md`）：

1. **无实时视频流**——相机只能 `takePhoto` 取单帧。→ 状态输入只能是**引导式逐面拍照 + 人工确认**，做不了实时识别。
2. **单色绿显示、无 AR 配准**——箭头无法贴在物理魔方上。→ 用**固定持握参照系 + 面字母（U/D/L/R/F/B）+ 旋转箭头**传达指令，不依赖颜色。
3. **单轴交互**——硬件只有**向前 / 向后 / 单击 / 双击**，无四方向键、不能任意点选。映射：向前/向后移动焦点、**单击**=确认/切换、**双击**=返回上一级（系统级）。→ 一切前进操作都要能用"逐项移动 + 单击"走到。

---

## 3. 功能范围

**做（In Scope）**
- 引导式六面拍照扫描，自动识别每面 9 格颜色。
- 逐面确认纠错：手动（选格切色）+ 语音（"白色面下方应该是蓝色"）两种方式。
- 状态校验：非法时给出具体原因（如"白色 7 个、红色 10 个"）。
- 近最优分步求解（Kociemba 两阶段，~20 步）。
- HUD 单步引导：一次只显示一步（箭头 + 转动面 + 进度）。
- 语音口令推进（下一步 / 上一步 / 重复）+ 语音播报方向。
- 完成反馈（步数统计）。

**不做（Out of Scope）**
- 实时识别、连续追踪、AR 配准——硬件不支持。
- 三面一拍——已验证放弃（广角下魔方太小、斜拍透视+反光，识别不可靠）。
- 计时排行、2/4/5 阶、多人社交。

---

## 4. 核心流程与交互

```
扫描(逐面拍照+纠错) → 首页核对+校验 → 求解 → 分步引导 → 完成
```

1. **扫描**：逐面提示对准 → 单击拍照 → 识别 9 格 → 手动或语音纠错 → 接受进入下一面（顺序 顶→右→前→底→左→后，固定持握）。
2. **核对校验**：六面回显首页，可继续逐格改色；单击「求解」。合法进入求解，非法显示具体原因就地修正。
3. **求解**：本地算出转动序列。
4. **分步引导**：HUD 显示当前一步（箭头+面字母+进度）；向前/单击 或 语音"下一步"推进，向后 或 "上一步"回退，"重复"再演示。
5. **完成**：提示 + 步数统计。

**交互模型**：单轴线性导航——向前/向后逐项移动焦点，单击确认/切换，双击返回上一级；所有按钮和色块都是可逐个到达的选项（前进操作不依赖双击）。

---

## 5. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **求解算法** | Kociemba 两阶段**近最优解（~20 步）** | 新手最在意快点还原成功；CFOP/LBL 步数是其 2–6 倍，太长劝退。详见 §6、附录 A |
| **求解实现** | 离线 **min2phase**，**免烘焙**（启动/首解建表一次 + 「初始化中…」） | 实测建表仅 ~1–4s 端侧一次性、内存 +3MB、求解 <1ms；比烘焙二进制简单。已接入 `solver.js` |
| **状态输入** | 引导式逐面拍照 + 手动/语音纠错 | 无视频流只能拍单帧；单帧难免误读，靠逐格纠错兜底（比整面重拍高效） |
| **指令显示** | 固定持握参照系 + 面字母 + 旋转箭头 | 单色无法表达颜色、无 AR 无法贴物理魔方；固定参照系是"哪个面"不歧义的唯一手段 |
| **教学模式** | 近最优为默认；层先法（LBL）分阶段教学为**可选**模式 | 默认重在快速还原；想理解原理时再切 LBL（规划，见附录 A） |

## 附录 A · 求解器方案对比

| 方案 | 步数 | 端侧负担 | 本硬件可行性 |
|------|------|---------|------------|
| ~~CFOP 表-free（`rubiks-cube-solver`）~~ | 50–130 | ~0 | 已弃用（步数太长） |
| **离线 min2phase（纯端侧，已采用）** | ~20 | 建表 ~1–4s 一次性 + 内存 3MB + 求解 <1ms | ✅✅ **已接入**：免烘焙，启动/首解建表一次 |
| 云端 Kociemba（fetch/WS） | ~20 | ~0 | 以后可选/加速：需联网（眼镜走蓝牙→手机链路） |
| 运行时建表 Kociemba（cubejs） | ~20 | 建表冻屏 | ❌ 当初卡死的方案 |
| WASM 两阶段 / Korf 最优 | ~20 / ≤20 | — | ❌ 无 WASM / 模式库过大 |

> min2phase 即 Kociemba 两阶段算法的一个优化实现（二者指同一方法，文中混用）。
> 不采用真·最优（Korf）：需 GB 级模式库 + 长搜索，本硬件不可行。
> **层先法（LBL）可选教学模式**：按七阶段（白十字→白角→第二层→黄十字→顶黄→顶角→顶棱）分组，每段先讲目标再给转动，面向想理解原理的用户。

## 附录 B · HUD 单步引导画面

```
┌──────────────────┐
│        12/48     │   ← 进度
│      ┌───┐       │
│   ↺  │ R │       │   ← 转动面字母 + 旋转箭头(↻顺/↺逆，对着该面看)
│      └───┘       │
│   "右面 逆时针"   │   ← 文字 / 语音同步
└──────────────────┘
```
原则：一次一步、箭头优先、不显示下一步预览；扫描回显用中文色名（白/黄/红/橙/绿/蓝）。

## 附录 C · 语音纠错话术规范

触发：在扫描确认页（识别结果回显时）直接说出即可，系统监听并把"面+方位"映射到对应格、改为目标色。
格式：「**[某色]面 + [方位] + 应该是 [目标色]**」，如"白色面下方应该是蓝色""红色面左上方应该是绿色"。
- 方位词（对应 3×3 九格）：`左上 / 上 / 右上` · `左 / 中 / 右` · `左下 / 下 / 右下`。
- 面的指代：颜色名（白色面…，按中心块）或方位名（顶/前/右/底/左/后）皆可。
- 面/方位/颜色任一缺失或歧义 → 语音回询，不盲改。

## 附录 E · 颜色识别管线

1. `<camera>` 预览上叠 3×3 引导框（中央 66%），提示填满框。
2. 单击拍照 → 解码 RGBA → 9 格中心小块采样平均色。
3. RGB→HSV，按色相阈值分 6 色（白色按低饱和+高亮度判定）。
4. 六面集齐组装网格，中心格强制身份色抗误读。
> 阈值集中在 `lib/colors.js#CONFIG`，待真机光照校准；环境光影响白/黄、橙/红区分（靠逐格纠错+非法校验兜底）。
