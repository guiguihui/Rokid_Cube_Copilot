# 魔方还原助手 · 开发实现方案

> 平台：AIUI（Rokid Glasses，Ink/QuickJS） · 版本：v2.2 · 更新：2026-06-17
> 配套：产品决策见 `魔方还原助手_MVP_产品设计.md`；API 写法见 `硬件与API开发参考.md`；逐次改动见 `开发状态.md`。
> **本文是 TDD 执行清单**：每个任务 = 改哪个文件 + 依赖 + **测试先行(开发中)** + **完成验收(开发完)**。**W1、W2 是项目核心**，优先且最严。

## 1. 现状基线（从哪接手）

MVP 闭环 `扫描 → 核对校验 → 求解 → 分步引导 → 完成` 已可在 Craft/真机运行，当前 `v1.0.13`。应用在仓库根 `E:\ROKIT`，`app.json` 三页 `input→scan→guide`。

**已完成、勿重写**：项目结构/路由/`wx` 跨页存储；首页(六面展示+逐格改色+具体校验)；扫描(拍照识别+逐格手动纠错)；webp/png/jpeg 分流解码；引导(Canvas 箭头+步数+语音)；单轴交互+滑动节流；全中文显示。

**`lib/` 现成模块**（在此基础上改）：
| 文件 | 现有能力 |
|------|---------|
| `cube.js` | 颜色↔面映射、中英文名表、`solvedGrid`、`gridToFacelet`/`faceletToGrid`(min2phase 用,URFDLB)、`validate`(具体原因)、`nextColor`、`SCRAMBLE_GRID`/`SCRAMBLE_GRIDS`(多打乱态)、`cloneGrid`；`gridToRcs` 已废弃(旧 CFOP) |
| `solver.js` | `solve(grid)`(min2phase 近最优+归一化)、`scrambledGrid`(循环切多态)、`selfTest` |
| `vendor/min2phase.js` | Kociemba 两阶段求解器(已接入)；`vendor/cube-solver.js` 已废弃(死文件,可删) |
| `moves.js` | `parseMoves`/`stepText`(走步→中文步骤) |
| `colors.js` | `sampleFace`(中央 66% 3×3 采样)、`classifyColor`、阈值 `CONFIG` |
| 解码器 | `webp/png/jpeg.js`+`vendor/*` |

> 关键 API 写法（Canvas `wx.createCanvasContext`+`onShow`、跨页 `wx.setStorageSync`、单帧 `takePhoto`、绝对路由等）照 `硬件与API开发参考.md`，写代码前先查。

> **测试/调试辅助（非产品功能，发布前酌情移除）**：首页「载入测试打乱」(`scrambledGrid` 循环切 6 个态)、「重置」；扫描页「测试图片」(`lib/testImages.js` 绕过相机)、自诊断行 camhint；三页底部 debug 行(`lastKey`/焦点/中心RGB)。
> **交互返回**：硬件"双击=返回上一级"已**真机确认全界面系统级生效**，App 无需处理双击事件。`Backspace` 分支仅作 PC/键盘调试兜底（首页=退出小程序、scan/guide=返回上一页）。

## 2. 开发范式：TDD + 双重验收

- **纯逻辑模块**（`cube.js`/`solver.js`/`moves.js`/`colors.js`/`voicefix.js` —— 无 `wx` 依赖、可在 node 直接 `import`）：**先写测试，再写实现**。测试放 `tests/*.test.mjs`，`node --test tests/` 跑；红(失败)→绿(实现到全过)→重构。本项目逻辑层一向 node 可测，沿用。
- **UI/设备相关**（页面渲染/相机/语音/真机手感）：node 测不了，**先定"可观察验收清单"再动手**，在 Craft/真机按清单逐条核对。能抽出的纯函数（如降采样、解析）仍走 node 测。
- **每个任务两组标准**：
  - **测试先行（开发中）**：动手前先落下的测试用例或观察清单——它**定义"做完"长什么样**，边写边对着它从红到绿。
  - **完成验收（开发完）**：全部测试绿 + 集成/端到端通过 + 无回归。
- 卡片格式：**改动** · **依赖** · **测试先行(开发中)** · **完成验收(开发完)**。

## 3. 任务分解

### ★ W1 · 求解内核升级为近最优（核心，最高优先）

> 目标：解从 50–130 步降到 ~20 步（Kociemba 两阶段）。
> **进度（2026-06-18）**：T1.1–T1.4a、T1.6（A+B）**已完成**（min2phase 唯一求解路径、TDD 10/10 绿、19 步、首页预热+求解中提示）。
> **已定方案：离线 min2phase（`lib/vendor/min2phase.js`），免烘焙二进制**——实测建表仅 ~1–4s 端侧一次性、内存 +3MB、稳态求解 <1ms。**下一步**：T1.5 真机量首解/预热实际冻屏（验证体感），随后并行 W2/W3。

**T1.1 测量原型（node，拿决策数据）**（✅ 已完成）
- 已做：`tools/measure-kociemba.cjs`（cube-solver 慢代理，含正确性闸门 50/50）、`tools/measure-min2phase.cjs`（cstimer 真 min2phase）、`tools/measure-min2phase-build.mjs`（vendored 建表/内存/求解）。
- 实测结论：真 min2phase 步数 20、稳态求解 <1ms、内存 +3MB → 数据支持离线（详见 `开发状态.md`）。

**T1.2 决策门（离线 vs 云端）**（✅ 已决：离线 min2phase）
- 依据：min2phase 内存 +3MB、求解 <1ms、建表 ~1–4s 一次性 → 离线完全可行、断网可用、无云端长尾/联网依赖。云端降为以后可选。

**T1.3a 离线·vendor 源 + 验证**（✅ 已完成，免烘焙）
- 已做：`lib/vendor/min2phase.js`（cstimer 的 Shuang Chen min2phase，IIFE、无 node/浏览器依赖、尾加 `export default`）；`tests/min2phase.smoke.test.mjs`（已还原态 0 步 + 50 随机态往返还原 ≤25 步）**全绿**；`package.json` 加 `"type":"module"`。
- 实测：建表 ~105ms node（端侧 ~1–4s 一次性）、内存 +3MB、稳态求解 <1ms。
- 结论：**免烘焙二进制**，启动/首解建一次表 + 「初始化中…」提示即可。
- 待真机确认：一次性建表实际冻屏 <~5s（>5s 才回退到"`initFull` 后序列化烘焙"）。

**T1.4a 离线·接线**（✅ 已完成，TDD 全绿）
- 已做：`cube.js` 加 `gridToFacelet`/`faceletToGrid`；`solver.js#solve` 改走 min2phase（签名/输出不变、归一化空格、Error 抛错）；`tests/solver.test.mjs` 6 用例(先红后绿)。
- 验收达成：`node --test tests/` **8/8 PASS**；`selfTest` **19 步**；`moves.js` 解析 0 异常 → `input`/`guide` 不改即用。
- 待真机：Craft 端到端跑到完成（逻辑层已验证兼容）。

**T1.3b / T1.4b 云端分支**（若 T1.2=云端）
- 改动：接口契约 `POST {facelets:54字符} → {moves:"U R2 F' …"}`；`solver.js` 加 `solveRemote(grid)`，`fetch` + 超时 + 失败兜底（退回 CFOP 或提示联网）。
- 依赖：T1.2=云端。
- 测试先行：`tests/solver.remote.test.mjs`——mock `fetch`：正常返回→解析为走步串且格式合法；超时/错误→走兜底（不抛未捕获异常）。
- 完成验收：mock 测试绿；真机联网得 ~20 步；断网/超时有兜底不卡死。

**T1.5 端侧实测**
- 改动：Craft/真机量单次求解耗时，记入 `开发状态.md`。
- 依赖：T1.4a 或 T1.4b。
- 测试先行（阈值）：单次求解 **< ~2s** 视为通过。
- 完成验收：耗时达标；否则回 T1.2 重选分支。

**T1.6 首解初始化 UX**（min2phase 首次求解触发一次性建表 ~1–4s 同步冻屏，需消除"点了卡死"观感）
- 背景：现 `input#doSolve` 先 `setData({status:'solving'})` 再**同步** `solve()`，建表占满单线程 →「求解中…」来不及刷新。
- 改动：
  - **A 首解前先刷提示**（✅ 已完成 v1.0.15）：`doSolve` 把 `solve()` 推到 `setTimeout(…, 60)`，「⏳ 求解中…（首次需初始化，请稍候 1–4 秒）」先画出来再建表。
  - **B 预热(pre-warm)**（✅ 已完成 v1.0.16）：`solver.js` 加 `warmup()`/`isWarmed()`(幂等，建满表)；首页 `onLoad` 后 `setTimeout(300ms)` 预热，显示"⏳ 求解器初始化中…"。建表卡顿挪到首屏空档，之后求解秒出。
- 依赖：T1.4a；建议 T1.5 真机冻屏数据出来后定预热放哪（onLoad 延迟 / 进求解前）。
- 测试先行（清单）：① 冷启动后第一次点求解，**先看到"初始化中…"**再出结果，不是无反应；② 预热完成后再点求解，**无可感知等待**。
- 完成验收：真机首解不出现"点了没反应"的观感；二次求解即时。

### ★ W2 · 扫描可用性（核心：无实时预览的应对）

**T2.1 快照复核渲染**（✅ 已完成 v1.0.17）
- 已做：`colors.js#downsample(rgba,w,h,n,region?)` 纯函数（n×n 色块均值，region 取中央正方形=对齐采样区）；`pages/scan/index.ink` review 模式 `<camera>`↔`<canvas id="review">` 按 `inReview` 切换，拍照后 `drawReview()` 把缩略图画到画布（40×40 色块），HTML `.guide` 3×3 框叠在上面=采样落点。
- 测试：`tests/downsample.test.mjs` 2 用例（尺寸 n×n、区域均值）**绿**。
- 待真机：拍照后能否认出刚拍的面、框是否对齐（设备端 canvas 绘制 + 80ms 延迟是否够）。

**T2.2 复核交互闭环**
- 改动：缩略图旁保留逐格纠错 + 「接受/重拍」。
- 依赖：T2.1。
- 测试先行（清单）：取景偏→重拍重来本面；取景对→接受进下一面；六面齐→写 `cube.scannedGrid` 回首页。
- 完成验收：清单逐条在 Craft/真机过。

**T2.3 颜色识别：相对锚色（替代绝对阈值）**（✅ 分类器已完成 v1.0.19）
- **定位决策**：自动检测在杂乱广角场景不可靠（实测：饱和→框到手、色相多样性→框到笔记本屏幕诱饵），故**用引导框定位**（用户把魔方面正对填满框），避透视、稳。
- **相对识别**（已做）：`colors.js#classifyByAnchors(rgb, anchors)`——用本魔方 **6 个中心块作锚色**，其余块按最近锚色（色度去亮度距离）归类，全相对、自适应光照/不同配色。`scan` finalize 用 6 中心锚色统一识别非中心非手动格（手动改过的保留）。
- 测试：`tests/classify.test.mjs` 模拟白平衡+明暗+噪声——**锚色 avg 99.7%/worst 96.3% vs 绝对阈值 82.6%**。
- 待真机：用**引导框正对捕获**做实拍验证（`testImg/realImg` 那 6 张是未引导、魔方小且斜，需手动框，不适合直接验分类器）。

### W3 · 语音纠错（扫描确认页）

**T3.1 解析器（天然 TDD）**
- 改动：新建 `lib/voicefix.js`：`parseFix(text) → {face,index,color} | null`。颜色词→色字母、方位词→格 index(左上=0…右下=8)、面用颜色/方位名反查；任一缺失或歧义→`null`。
- 依赖：无。
- 测试先行：`tests/voicefix.test.mjs` 先写——
  `"白色面下方应该是蓝色"` → `{face:'U',index:7,color:'B'}`；
  `"红色面左上方应该是绿色"` → `{face:'R',index:0,color:'G'}`；
  方位词全表(左上0/上1/右上2/左3/中4/右5/左下6/下7/右下8)各一例；
  残缺(`"应该是蓝色"`)/歧义 → `null`。
- 完成验收：全部用例绿。

**T3.2 接线 scan review**
- 改动：review 模式启 `SpeechRecognition`；`onresult`→`parseFix`→命中改对应格、`null` 语音回询；`onend` 重启；不与翻页命令冲突。
- 依赖：T3.1。
- 测试先行（清单）：说示例话→改对应格；歧义→回询不乱改；与翻页命令共存不误触。
- 完成验收：真机清单逐条过。

## 4. 推进顺序（依赖与并行）

```
★W1: T1.1 → T1.2 → (T1.3a→T1.4a | T1.3b→T1.4b) → T1.5 → T1.6(首解 UX)  ← 核心，先行
★W2: T2.1 → T2.2 → T2.3                                   ┐ 核心，可与 W1 并行
 W3: T3.1 → T3.2                                          ┘（独立于求解器）
 W4 / W5 / W6：收尾
```
- **W1、W2 是核心**：优先排期、测试最严（W1 求解正确性、W2 识别准确率直接决定产品成立与否）。
- T1.1 拿到数据前不写离线/云端实现；W2/W3 与 W1 无耦合可并行；W4/W5/W6 收尾。
- **每个任务都先把"测试先行"那条落成 `tests/*.test.mjs` 或清单，再开始改实现。**
